# 前端控制面板第一版设计规格

日期：2026-05-29

## 1. 目标

第一版先完善网页控制面板的页面结构、操作流程、表格字段和前端交互，不急着做后端、微信小程序、苹果端、安卓端。

这个版本的重点不是简单跑通一条流程，而是把水果店老板每天真正会用到的前端功能先设计完整：

- 好录入：先选商铺，再记账。
- 好核对：语音识别结果必须能人工修改。
- 好查账：按日期、商铺、水果、月份都能看。
- 好收款：能看未付款、部分付款、已付款，并支持批量收款。
- 好追溯：价格、账单、付款、修改都要有记录。

## 2. 第一版范围

第一版包含五个页面：

1. 今日记账
2. 价格单
3. 账单汇总
4. 识别记录
5. 设置

第一版先使用前端假数据或本地模拟数据。页面和流程确认稳定后，再设计后台来保存真实数据。

## 3. 网页和小程序入口差异

### 3.1 网页控制面板

网页采用左侧固定菜单：

- 今日记账
- 价格单
- 账单汇总
- 识别记录
- 设置

点击左侧菜单后，右侧显示对应页面。打开网页后，默认进入今日记账。

### 3.2 小程序入口

小程序不使用左侧菜单。首页显示五个大按钮：

- 今日记账
- 价格单
- 账单汇总
- 识别记录
- 设置

点击按钮进入独立页面。每个独立页面都要有返回按钮。

### 3.3 页面一致性

网页和小程序的功能规则保持一致，只是入口形式不同。这样以后从网页迁移到小程序时，不需要重新设计业务流程。

## 4. 今日记账

### 4.1 页面目的

今日记账是老板每天最常用的页面，用来记录固定商铺送货、临时客户、散客或普通零散账。

### 4.2 页面入口

进入今日记账后，先显示商铺选择区。

商铺按钮排序：

1. 手动置顶商铺
2. 按本月销售金额从高到低排列的商铺
3. 普通记账
4. 添加商铺

普通记账固定显示，不参与本月销量排序。

商铺选择区同时提供草稿箱入口，用来查看还没有生成账单的草稿。

### 4.3 商铺和普通记账

商铺用于固定客户，比如王姐超市、东门水果摊。

普通记账用于没有固定商铺名称的零散单子，比如临时客户、散客、自己店内零散销售。

点击商铺或普通记账后，进入独立记账页面。

页面标题示例：

- 王姐超市 / 2026年5月29日
- 普通记账 / 2026年5月29日

### 4.4 日期规则

进入记账页面时，默认使用当前登录当天日期。

页面提供补记日期入口。选择非当天日期时，页面必须明显提示：

当前正在补记：2026年5月20日

避免老板误把旧账记到今天。

### 4.5 页面流程

今日记账页面实际操作分为两层：

1. 语音或文字录入区：老板点击大的录音按键后，系统自动把识别内容填入核对表格。
2. 核对和生成账单流程：老板只需要核对表格，再生成账单。

页面上的步骤条只保留两个主步骤：

1. 核对
2. 生成账单

不要再把录入做成步骤条里的一个步骤，因为大的录音按键已经承担了录入动作，二者重复。

### 4.6 录入区

录入方式：

- 第一版先用文字模拟语音识别结果。
- 后续再接真实语音识别。

录入区字段：

| 字段 | 说明 |
|---|---|
| 商铺 | 当前选择的商铺或普通记账 |
| 记账日期 | 默认当天，可补记 |
| 原始识别文字 | 语音识别后的文字，第一版可手动输入 |
| 备注 | 可选，比如客户明天结、少送一箱 |

主要按钮：

- 保存草稿
- 清空本次录入

清空本次录入会丢弃当前输入内容，执行前必须二次确认。

录入区规则：

- 录入区放在页面上方。
- 语音或文字识别成功后，识别结果自动填入下方核对表格。
- 不需要额外的开始核对按钮。
- 页面只保留大的录音按键，文案为点击录入，不再额外显示开始录入按钮。
- 模拟识别是第一版内部默认能力或开发测试能力，不作为老板需要选择的页面按钮展示。

### 4.7 核对区

系统把原始文字拆成表格，老板必须能人工修改。

核对区是今日记账页面的核心版面，必须比最近账单、价格提醒、统计摘要等辅助模块更大。文字可以根据空间做简短处理，但水果名称、数量、单位、单价、小计、状态和操作按钮必须清楚可读。

核对表格字段：

| 字段 | 说明 |
|---|---|
| 水果 | 可修改，可从价格单选择 |
| 规格 | 支持多规格，默认匹配常用规格 |
| 数量 | 可修改 |
| 单位 | 可修改 |
| 单价 | 从价格单按日期获取 |
| 单价来源日期 | 显示本次使用哪天生效的价格 |
| 小计 | 数量乘以单价 |
| 异常提示 | 无价格、单位不一致、疑似重复等 |

主要按钮：

- 添加一行
- 删除本次核对行
- 重新识别
- 保存草稿
- 生成账单

删除本次核对行只影响当前草稿，执行前要提示确认。

版面要求：

- 核对表格尽量横向铺开，不要被右侧摘要或最近账单挤得太窄。
- 核对表格要优先预留更多列，不要为了显得简洁而把字段压得太少。后续会有规格、价格来源、异常提示、备注等输入事项，表格需要提前留出位置。
- 网页端核对表格可以比示例图列更多，必要时使用横向空间、紧凑列宽或横向滚动；不要把重要字段藏到二级页面里。
- 推荐核对表格列：序号、水果名称、规格、数量、单位、单价、价格来源日期、小计、异常提示、备注、状态、操作。
- 表格底部固定显示添加一项、共几项、合计金额。
- 合计金额使用醒目的主色，位置靠右，方便老板核对总价。
- 行内操作按钮要清楚区分：修改用主色描边，删除用红色描边。
- 最近账单和价格匹配提醒属于辅助信息，不要占用整页空间。优先做成按键入口、折叠区、右侧窄栏或小提醒。
- 后续价格单、账单汇总、识别记录页面也遵守同一规则：核心表格放大，辅助信息收起。

按键风格要求：

- 分段按键和页面切换按钮参考用户确认过的第三种按键风格。
- 当前选中项使用深色底、白色文字、较强对比。
- 未选中项使用浅色底、细边框、深灰文字。
- 颜色可以微调，避免太冷或太跳。推荐主按钮使用深绿或深墨色，辅助按钮使用浅灰或浅绿。
- 不要让按键占满整页，按键用于切换、筛选、收起辅助信息，而不是替代核心表格。

### 4.8 价格异常提醒

以下情况必须提醒老板确认：

- 价格单没有这个水果。
- 水果有多个规格，但系统无法判断规格。
- 录入单位和价格单单位不一致。
- 同一商铺、同一日期、金额接近，可能重复记账。

第一版不自动换算单位。比如价格单是斤，录入是箱，系统只提醒确认，不擅自换算。

### 4.9 水果名称自动匹配

语音录入后，系统要尽量把识别出来的水果名称匹配到价格单里的标准水果名。

匹配顺序：

1. 先匹配价格单里的标准水果名。
2. 再匹配价格单里的水果别名。
3. 如果还是不完全一致，给出接近的水果候选项。

示例：

| 识别文字 | 系统建议 |
|---|---|
| 红富 | 红富士苹果 |
| 密桃 | 水蜜桃 |
| 香蕉蕉 | 香蕉 |
| 火龙 | 火龙果 |

规则：

- 如果系统能明确匹配，核对表格里直接显示标准水果名。
- 如果系统不确定，必须显示候选项，让老板人工选择。
- 不确定的水果不能直接生成账单。
- 老板确认过的匹配可以进入价格单别名候选，后续优先推荐。
- 第一版不要自动悄悄加入别名，必须由老板在价格单里确认后才正式加入别名。
- 最终账单里保存标准水果名，不保存容易混乱的识别错字。

### 4.10 生成账单

生成账单前显示确认信息：

| 字段 | 说明 |
|---|---|
| 商铺 | 当前商铺或普通记账 |
| 日期 | 账单日期 |
| 明细 | 水果、规格、数量、单位、单价、小计 |
| 总金额 | 所有小计合计 |
| 付款状态 | 默认未付款，可改为部分付款或已付款 |
| 备注 | 可选 |

如果账单内容已经提前确认，可以生成可打印送货单，送货时不用手写。

付款状态：

- 未付款
- 部分付款
- 已付款

生成账单后：

- 进入账单汇总。
- 保留识别记录。
- 账单锁定当时使用的单价。

### 4.11 打印送货单

今日记账支持打印送货单。

使用场景：

- 送货前，账单内容已经确认，可以先打印送货单。
- 送货时把纸质送货单给商铺核对或签字。
- 后续对账时，系统账单和纸质送货单能互相对应。

打印入口：

- 生成账单前：确认内容后，可以打印预览送货单。
- 生成账单后：账单详情里可以再次打印送货单。

送货单字段：

| 字段 | 说明 |
|---|---|
| 门店名称 | 当前水果店名称 |
| 商铺 | 商铺名称或普通记账 |
| 送货日期 | 账单日期 |
| 账单编号 | 方便和系统账单对应 |
| 水果明细 | 水果、规格、数量、单位、单价、小计 |
| 调整项 | 退货、少送、补送，如有 |
| 合计金额 | 本单总金额 |
| 备注 | 可选 |
| 收货确认 | 预留签字或确认位置 |

打印规则：

- 第一版只做浏览器打印预览，不接打印机硬件。
- 打印送货单不等于发送消息，不自动发给外部人员。
- 如果后续要通过微信、短信、邮件发送送货单，必须先让用户确认。
- 打印版要干净，不显示左侧菜单、筛选器、编辑按钮等后台操作内容。

### 4.12 账单生成后的修改

账单生成后，如果发现水果、规格、数量、单位、单价或备注有错误，允许修改。

修改规则：

- 修改前必须显示原账单内容，避免误改。
- 修改后重新计算账单金额。
- 每次修改都要留下修改记录。
- 修改记录里要保存修改时间、修改前内容、修改后内容和修改原因。

修改记录示例：

| 字段 | 说明 |
|---|---|
| 修改时间 | 什么时候修改 |
| 修改字段 | 比如数量、单价、付款状态 |
| 修改前 | 原来的内容 |
| 修改后 | 新内容 |
| 修改原因 | 必填，比如录入错误、客户确认少送 |

### 4.13 退货、少送、补送

第一版支持简单的账单调整项，用来处理退货、少送、补送。

这个功能只影响账单金额和对账，不做复杂库存。

调整项第一版挂在原账单下面，不单独新建一张调整账单。调整项会影响原账单总金额、商铺对账单和水果统计。

调整项类型：

- 退货
- 少送
- 补送

调整项字段：

| 字段 | 说明 |
|---|---|
| 类型 | 退货、少送、补送 |
| 水果 | 对应水果 |
| 规格 | 对应规格 |
| 数量 | 可正可负 |
| 单位 | 斤、千克、个、盒、袋、件、箱等 |
| 单价 | 使用账单日期对应价格，或手动确认 |
| 金额 | 正数或负数 |
| 原因备注 | 必填 |

示例：

- 退苹果1箱，金额 -120元。
- 香蕉少送5斤，金额 -20元。
- 补送香蕉5斤，金额 +20元。

### 4.13 草稿

今日记账支持草稿。老板录入到一半可以先保存，之后继续核对。

今日记账页面需要有草稿箱。草稿箱显示所有未完成草稿，老板可以按商铺、日期快速找回。

草稿字段：

| 字段 | 说明 |
|---|---|
| 草稿编号 | 草稿唯一编号 |
| 草稿时间 | 保存草稿的时间 |
| 商铺 | 当前商铺或普通记账 |
| 记账日期 | 账单日期 |
| 原始文字 | 输入的识别文字 |
| 核对明细 | 已拆分或已修改的表格 |
| 备注 | 可选 |

草稿箱表格字段：

| 字段 | 说明 |
|---|---|
| 保存时间 | 草稿保存时间 |
| 商铺 | 商铺或普通记账 |
| 记账日期 | 草稿对应账单日期 |
| 原始文字摘要 | 显示前一小段，方便识别 |
| 状态 | 未核对、核对中 |
| 操作 | 继续核对、作废草稿 |

草稿不直接删除，第一版使用作废草稿。

## 5. 价格单

### 5.1 页面目的

价格单是全局统一价格来源。今日记账和账单生成必须从价格单取价，不能在不同地方各自维护价格。

### 5.2 多规格

同一种水果可以有多个规格。

示例：

| 水果 | 规格 | 当前单价 | 单位 |
|---|---|---:|---|
| 苹果 | 普通 | 5 | 斤 |
| 苹果 | 精品 | 8 | 斤 |
| 苹果 | 箱装 | 120 | 箱 |

今日记账时默认匹配常用规格，识别不准时老板可以手动选择其他规格。

### 5.3 价格生效日期

价格按日期生效。

规则：

- 如果某天更新价格，从生效日期开始使用新价格。
- 如果某天没有更新价格，就沿用之前最近一次价格。
- 生成账单时，按账单日期取当时有效价格。
- 旧账单不跟着价格单变化。

示例：

- 5月1日：苹果普通 5元/斤
- 5月29日：苹果普通 6元/斤
- 补记5月20日账单时，用5元/斤
- 记5月29日账单时，用6元/斤

### 5.4 价格单表格字段

| 字段 | 说明 |
|---|---|
| 水果名称 | 比如苹果、香蕉 |
| 别名 | 比如红富士、富士苹果，可帮助识别匹配 |
| 规格 | 比如普通、精品、箱装 |
| 当前单价 | 当前有效价格 |
| 单位 | 斤、千克、个、盒、袋、件、箱等 |
| 生效日期 | 当前价格从哪天开始生效 |
| 是否启用 | 停用后不在今日记账默认匹配中出现 |
| 常用规格 | 今日记账优先匹配 |
| 备注 | 可选 |

主要按钮：

- 新增水果
- 新增规格
- 修改价格
- 查看历史价格
- 停用

价格不建议删除，只能停用。删除价格会影响查账。

### 5.5 水果别名

价格单支持水果别名，用来处理语音识别和老板口头叫法不同的问题。

示例：

| 标准水果名 | 别名 |
|---|---|
| 苹果 | 红富士、富士苹果、苹果果 |
| 火龙果 | 红心火龙果、白心火龙果 |
| 水蜜桃 | 蜜桃、桃子 |

识别规则：

- 今日记账识别到别名时，优先匹配到标准水果名。
- 如果一个别名可能对应多个水果，必须提醒老板人工选择。
- 别名只帮助匹配，不改变账单里最终显示的标准水果名。
- 识别结果接近但不完全一致时，可以结合别名和相似名称给出候选项。

别名候选规则：

- 今日记账里老板确认过的接近名称，可以进入别名候选。
- 价格单页面提供别名候选列表。
- 老板确认后，候选别名才正式加入对应水果。
- 老板拒绝后，该候选不再优先提示。

### 5.6 历史价格记录

每次改价都保留记录。

历史价格字段：

| 字段 | 说明 |
|---|---|
| 水果 | 水果名称 |
| 规格 | 规格名称 |
| 单价 | 当时设置的价格 |
| 单位 | 价格单位 |
| 生效日期 | 从哪天开始有效 |
| 修改时间 | 什么时候修改 |
| 备注 | 可选 |

## 6. 账单汇总

### 6.1 页面目的

账单汇总用于查账、对账、收款、导出。

第一版必须同时支持：

- 按天查看
- 按商铺查看
- 按水果统计
- 按月统计汇总

### 6.2 顶部筛选

顶部筛选项：

| 筛选项 | 说明 |
|---|---|
| 时间范围 | 今日、昨日、本月、自选日期、自选月份 |
| 商铺 | 全部商铺、某个商铺、普通记账 |
| 付款状态 | 全部、未付款、部分付款、已付款 |
| 对账状态 | 全部、未对账、已对账 |

### 6.3 汇总卡片

页面顶部显示关键数字：

### 6.8 付款状态修改

付款状态按整张账单修改，不按每个水果明细修改。

状态：

- 未付款
- 部分付款
- 已付款

修改付款状态时要记录：

| 字段 | 说明 |
|---|---|
| 修改时间 | 什么时候改 |
| 原状态 | 修改前状态 |
| 新状态 | 修改后状态 |
| 本次收款金额 | 如果有收款，记录金额 |
| 备注 | 可选，比如微信收款、现金 |

### 6.9 批量收款

账单汇总支持批量收款。

操作方式：

1. 筛选未付款或部分付款。
2. 勾选同一个商铺的多张账单。
3. 点击批量收款。
4. 弹窗确认收款信息。

批量收款弹窗字段：

| 字段 | 说明 |
|---|---|
| 商铺 | 当前收款商铺 |
| 勾选账单数量 | 本次收款涉及几张账单 |
| 应收总金额 | 勾选账单待收合计 |
| 本次收款金额 | 默认等于应收总金额，第一版不允许填小于应收总金额 |
| 付款方式备注 | 可选，比如现金、微信、转账 |

规则：

- 第一版批量收款只做全额批量结清。
- 全额确认后，选中账单改为已付款。
- 本次收款金额默认等于应收总金额，不允许输入小于应收总金额。
- 如果本次只收到部分金额，不自动分配到多张账单。
- 如果少收，页面提示进入商铺详情或单张账单，手动填写每张账单收到多少。
- 第一版不做自动抵扣最早欠款，避免系统分配规则不清楚。

账单详情字段：

| 调整项 | 退货、少送、补送 |
| 总金额 | 明细和调整项合计后的金额 |
| 付款记录 | 每次收款时间、金额、备注 |
| 修改记录 | 每次修改前后内容和原因 |

已付款不等于已对账。付款状态表示钱收没收，对账状态表示双方有没有确认这笔账。

- 作废账单仍可查看。
- 作废时必须填写原因。
- 作废前必须二次确认。

## 7. 识别记录

### 7.1 页面目的

识别记录用于保留原始语音文字、修正后的结果，以及它们是否生成了账单。

这样以后查账时，可以知道某张账单最初是怎么识别出来的。

### 7.2 表格字段

| 字段 | 说明 |
|---|---|
| 识别时间 | 什么时候识别 |
| 商铺 | 商铺或普通记账 |
| 记账日期 | 账单日期 |
| 原始文字 | 语音识别出的原文 |
| 修正结果 | 人工修改后的明细 |
| 状态 | 未生成账单、已生成账单、已作废 |
| 生成账单 | 对应账单编号 |
| 操作 | 查看、继续核对、复制成新草稿、作废 |

### 7.3 状态

识别记录状态：

- 未生成账单
- 已生成账单
- 已作废

### 7.4 操作规则

未生成账单的识别记录：

- 可以继续核对。
- 可以保存草稿。
- 可以作废。

已生成账单的识别记录：

- 可以查看。
- 可以复制成新草稿。
- 不能直接覆盖旧账单。

如果旧账单确实有错，应该在账单详情里修改或作废，而不是从识别记录直接改旧账。

复制成新草稿规则：

- 原账单不变。
- 新草稿进入今日记账的核对步骤。
- 新草稿可以重新生成新账单。

## 8. 设置

### 8.1 页面目的

设置页面用于管理门店信息、商铺、常用单位和默认记账规则。

### 8.2 门店信息

字段：

| 字段 | 说明 |
|---|---|
| 门店名称 | 当前水果店名称 |
| 联系方式 | 可选 |

### 8.3 商铺管理

字段：

| 字段 | 说明 |
|---|---|
| 商铺名称 | 比如王姐超市 |
| 联系人 | 可选 |

## 10. 第一版暂不做

为了避免范围失控，第一版暂不做：

- 真实后端数据库
- 真实语音识别接入
- 微信小程序真实开发
- 苹果端、安卓端
- 多门店切换
- 自动单位换算
- 自动抵扣最早欠款
- 自动发送微信、短信、邮件
- 复杂权限管理
- 线上付款或扣费功能

如果后续涉及授权第三方账号、发送外部消息、付款扣费、删除数据、清空数据库等操作，必须先提醒用户确认。

## 11. 交给 Claude Code 的实现边界

Claude Code 第一阶段只负责实现前端控制面板。

实现要求：

1. 使用前端假数据或本地模拟数据。
2. 必须实现五个页面入口。
3. 今日记账必须有商铺选择、日期、录入、核对、生成账单流程。
4. 今日记账必须有草稿箱，可以继续核对未完成草稿。
5. 今日记账必须能把语音识别文字里的水果名称匹配到价格单标准名称；不确定时必须人工确认。
6. 价格单必须支持多规格、当前价格、生效日期、历史价格展示和水果别名。
7. 账单汇总必须支持日汇总、月汇总、按商铺、按水果、付款状态、批量收款。
8. 账单必须有固定编号，生成后不随修改变化。
9. 今日记账和账单详情必须支持打印送货单预览。
10. 识别记录必须支持查看、继续核对、复制成新草稿。
11. 设置必须支持门店信息、商铺管理、单位管理、默认规则。
12. 账单不能直接删除，只能作废。
13. 付款状态和金额修改必须留下修改记录。
14. 页面要适合非程序员使用，按钮要大，表格要清楚，提示要直白。

交付后必须验证：

1. 页面能打开。
2. 左侧菜单能切换五个页面。
3. 今日记账可以用文字模拟语音结果。
4. 核对表格可以修改水果、规格、数量、单位。
5. 账单金额能用样例数据算对。
6. 价格按账单日期取价。
7. 旧账单不受新价格影响。
8. 账单汇总能按日、按月、按商铺、按水果查看。
9. 未付款和部分付款账单能勾选后全额批量结清；部分收款必须进单张账单手动处理。
10. 作废账单不参与统计，但能查看。
11. 草稿箱可以找回未完成草稿。
12. 水果别名能辅助匹配到标准水果名。
13. 接近水果名会给候选项，不确定时不能直接生成账单。
14. 送货单可以打开打印预览，打印内容不包含后台操作按钮。

## 12. 设计自查

- 没有要求第一版做后端，符合先完善前端控制面板的方向。
- 五个页面都有页面目的、主要字段和关键操作。
- 价格规则明确：按日期生效，账单锁定当时单价。
- 付款规则明确：按账单修改，支持批量收款。
- 删除风险已规避：账单、商铺、价格都不建议删除，使用作废或停用。
- 小程序只定义入口差异，不进入真实开发。
- 后续后台需要记录的数据已提前列出，但不要求第一版实现后台。
