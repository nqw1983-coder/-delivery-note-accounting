# 手写送货单网页控制面板架构规格

日期：2026-05-29

## 1. 目标

第一阶段先完善网页版控制面板架构，让水果店老板可以在 Mac 浏览器和苹果手机浏览器中完成手写送货单记账。

本阶段只围绕一个核心目标：上传或扫描手写送货单图片后，识别店面、日期、总金额，并填入对应年份、月份、日期和店面格子。

本阶段不做语音记账，不做完整后端，不做微信小程序，不做苹果端和安卓端。

## 2. 本规格优先级

本规格按当前最新交接说明整理，用来替代旧规格中语音记账、五个菜单页面、付款管理、价格单、账单汇总等第一版内容。

如果旧规格和本规格冲突，第一阶段按本规格执行。

主要收束点：

1. 左侧只放月份列表，不放今日记账、价格单、账单汇总、识别记录、设置五个菜单。
2. 核心页面是月表，不是语音记账页面。
3. 第一版只识别店面、日期、总金额，不识别水果明细。
4. 第一版不做付款、价格单、打印、导出、消息发送。
5. 第一版不保存原图文件本身，也不把查看原图作为第一版功能。

## 3. 第一版范围

第一版必须包含：

1. 左侧月份列表。
2. 年份选择，先固定 2026、2027、2028。
3. 搜索月份。
4. 增加月份按钮。
5. 年度统计按钮和弹窗。
6. 当前月份标题。
7. 上传文件框。
8. 扫描图片按钮。
9. 月份金额表格。
10. 单元格明细弹窗或右侧详情。
11. 识别结果确认区。
12. 手动修改店面、日期、金额和备注。
13. 疑似重复提示。
14. 新店面确认后新增列。
15. 待处理提示，但不在左侧增加入口。
16. 初始月份固定生成 2026 年 5 月、4 月、3 月、2 月、1 月，5 月排在最上面。

第一版暂不包含：

1. 真实后端数据库。
2. 真实图片识别服务接入。
3. 保存原图文件本身。
4. 水果明细识别。
5. 付款状态。
6. 价格单管理。
7. 发送消息、邮件或通知。
8. 删除数据、清空月份数据。
9. 导出、打印。

## 4. 控制面板整体结构

网页版采用左右结构：

| 区域 | 内容 |
|---|---|
| 左侧栏 | 标题、年度统计按钮、增加月份按钮、年份选择、搜索月份、月份列表 |
| 右侧顶部 | 当前月份标题、上传文件框、扫描图片按钮、待处理提示 |
| 右侧主体 | 当前月份金额表格 |
| 弹窗或右侧详情 | 单据识别结果、单元格明细、年度统计、新店面确认、疑似重复确认 |

左侧栏必须保持简单。除了月份相关内容，不增加其他业务入口。

## 5. 左侧栏设计

### 5.1 顶部标题区

显示：

1. 送货单记账
2. 年度统计按钮
3. 增加月份按钮

年度统计按钮打开年度统计弹窗。

增加月份按钮用于新增当前选择年份的下一个月份。

### 5.2 年份选择

第一版年份固定为：

1. 2026
2. 2027
3. 2028

选择年份后，月份列表只显示该年份下的月份。

年度统计也只统计当前选择年份。

### 5.3 搜索月份

搜索框用于快速找到月份。

示例：

1. 输入 5，可找到 2026年5月。
2. 输入 2026，可显示 2026 年所有已创建月份。
3. 输入 2026-05 或 2026年5月，可找到对应月份。

### 5.4 月份列表

规则：

1. 初始进入时，左侧先显示 2026 年 5 月、4 月、3 月、2 月、1 月。
2. 2026 年 5 月排在最上面，并默认选中。
3. 后续新增月份时，最新月份排在最上面。
4. 最多直接显示 12 个月。
5. 超出可滑动查看。
6. 当前选中的月份要有明显状态。
7. 月份项显示当月总金额和待处理数量。

月份项建议显示：

| 内容 | 示例 |
|---|---|
| 月份 | 2026年5月 |
| 总金额 | ￥12,860 |
| 待处理 | 待补 2 |

待处理只作为月份项或顶部提示，不单独做左侧入口，也不做点开列表。

## 6. 增加月份规则

初始月份不靠用户点击生成，系统默认准备好：

1. 2026年5月
2. 2026年4月
3. 2026年3月
4. 2026年2月
5. 2026年1月

点击增加月份按钮后：

1. 如果当前年份没有月份，新增 1 月。
2. 如果当前年份已有月份，新增该年份中最大月份的下一个月。
3. 2026 年初始最大月份是 5 月，所以第一次点击增加月份时新增 2026 年 6 月。
4. 如果已经有 12 月，再点击时提示当前年份已满 12 个月。
5. 新增月份排到月份列表最上面。
6. 新月份自动带出默认店面列。

默认店面：

1. 吾湘
2. 吾黄
3. 醉得意
4. 萍姐
5. 柳

## 7. 右侧顶部操作区

右侧顶部显示当前月份和扫描入口。

示例：

2026年5月

操作区包含：

1. 上传文件框。
2. 扫描图片按钮。
3. 待处理提示。

待处理提示示例：

有 2 张单据需要补充信息

如果待处理数量为 0，可以显示全部已处理或不显示提示。

第一版待处理保持简单，只显示数量或一句提示，不提供点开列表。

上传和扫描关系：

1. 用户先选择或拖入图片。
2. 点击扫描图片按钮。
3. 系统生成识别结果。
4. 识别清楚则自动进入对应月表。
5. 识别不清楚则进入确认区，由用户补充。

## 8. 月份金额表格

### 8.1 表格形态

表格左侧是日期，顶部是店面名称，中间格子是金额。

示例：

| 日期 | 吾湘 | 吾黄 | 醉得意 | 萍姐 | 柳 | 当日合计 |
|---|---:|---:|---:|---:|---:|---:|
| 1日 | 0 | 0 | 0 | 0 | 0 | 0 |
| 2日 | 52 | 0 | 165 | 0 | 0 | 217 |
| 3日 | 0 | 80 | 0 | 0 | 0 | 80 |

### 8.2 表格规则

1. 每个页面只显示一个月份。
2. 日期行按 1 日到月底排列。
3. 店面列按默认店面加新增店面排列。
4. 每个格子默认显示 0。
5. 同一天同店面多张单，格子显示总金额。
6. 当日合计自动汇总该日期所有店面金额。
7. 月合计在表格底部显示。
8. 金额右对齐，方便核对。
9. 移动端可以横向滑动表格，不能挤到看不清。

### 8.3 单元格点击

点击任意金额格子，打开该日期和店面的单据明细。

明细只显示加法来源，不做复杂账单页。

示例：

吾湘 / 2026年5月2日

52 + 165 = 217

明细字段：

| 字段 | 说明 |
|---|---|
| 序号 | 第几张单 |
| 金额 | 单据金额 |
| 识别状态 | 已确认、待补充、疑似重复 |
| 备注 | 用户填写 |
| 操作 | 修改 |

第一版修改金额后只保留最新金额，页面不显示最后修改时间。

## 9. 扫描识别流程

### 9.1 识别字段

第一版只识别：

1. 店面
2. 日期
3. 总金额

不识别水果明细。

### 9.2 识别优先级

1. 先识别日期。
2. 再识别总金额。
3. 再识别店面或客户名。

### 9.3 自动填写规则

只有以下信息都清楚时，才自动填入表格：

1. 日期清楚。
2. 金额清楚。
3. 店面能匹配已有店面，或用户已确认新增店面。

识别不清楚时：

1. 日期留空。
2. 金额留空。
3. 店面留空或进入候选。
4. 提示用户人工补充。
5. 不允许系统猜测并直接填入。

### 9.4 没有日期的处理

如果没有日期：

1. 不自动归入具体日期。
2. 在当前月份顶部显示待处理提示。
3. 用户补充日期后，再填入表格。

如果补充的日期不属于当前月份：

1. 提示该单据属于另一个月份。
2. 如果对应月份已经存在，用户确认后写入对应月份。
3. 如果对应月份不存在，第一版不自动创建，也不写入；提示用户先用增加月份按钮把月份建出来，再处理这张单。

这里涉及跨月份写入，会影响当前页面以外的月份数据，执行前必须提示用户确认。

### 9.5 没有金额的处理

如果没有金额：

1. 不生成金额。
2. 保留为空。
3. 提示用户填写金额。

金额填写后才能进入表格。

### 9.6 没有店面的处理

如果没有店面：

1. 不自动选择店面。
2. 提示用户选择已有店面或输入新店面。

如果识别到新店面：

1. 弹出确认。
2. 用户确认后，新增当前月份店面列。
3. 用户确认过的新店面，以后新增月份时默认带出。

## 10. 疑似重复规则

当满足以下情况时，提示可能重复：

1. 同一天。
2. 同店面。
3. 同金额。

第一版先按同金额判断，不做金额接近算法。

提示内容要简单：

这张单和已有记录很像，可能重复。确认后才加入。

用户选择：

1. 确认加入。
2. 返回修改。
3. 取消本次识别结果。

取消本次识别结果会丢弃当前未入账信息，执行前要提示确认。

## 11. 识别结果确认区

识别后显示一个确认区，可以是弹窗或右侧详情。

字段：

| 字段 | 规则 |
|---|---|
| 店面 | 可选择已有店面，可输入新店面 |
| 日期 | 可修改 |
| 金额 | 可修改 |
| 备注 | 可选 |
| 状态 | 已识别、待补充、疑似重复、已入账 |

按钮：

1. 确认入账。
2. 保存为待处理。
3. 取消本次结果。

确认入账前，如果会新增店面、跨月份写入或疑似重复，必须先弹出确认。

## 12. 年度统计

点击年度统计按钮后打开弹窗。

统计当前选择年份每个月总金额。

字段：

| 月份 | 总金额 | 待处理数量 |
|---|---:|---:|
| 1月 | 0 | 0 |
| 2月 | 0 | 0 |
| 3月 | 0 | 0 |

规则：

1. 每个月总金额来自该月份表格合计。
2. 第一版只统计当前选择年份。
3. 点击某个月可以切换到对应月份。

## 13. 数据结构草案

第一版可以先用前端模拟数据，但字段要按后续保存真实数据来设计。

### 13.1 月份

| 字段 | 说明 |
|---|---|
| year | 年份 |
| month | 月份 |
| stores | 本月店面列 |
| entries | 本月单据记录 |
| createdAt | 创建时间 |
| updatedAt | 修改时间 |

### 13.2 单据记录

| 字段 | 说明 |
|---|---|
| id | 单据编号 |
| storeName | 店面 |
| date | 日期 |
| year | 年份 |
| month | 月份 |
| amount | 总金额 |
| remark | 备注 |
| status | 已入账、待补充、疑似重复 |
| recognition | 自动识别结果 |
| corrected | 人工确认后的结果 |
| createdAt | 创建时间 |
| updatedAt | 系统内部更新时间，第一版页面不显示 |

### 13.3 识别结果

| 字段 | 说明 |
|---|---|
| rawStoreName | 识别出的店面 |
| rawDate | 识别出的日期 |
| rawAmount | 识别出的金额 |
| confidence | 简化可信度 |
| unclearFields | 不清楚字段 |

第一版不保存原图文件本身，单据记录里也不要求保存 imageName 或 imagePreview 字段。

## 14. 页面状态

页面至少要覆盖这些状态：

1. 初始进入时：默认打开 2026 年 5 月。
2. 没有月份时：提示点击增加月份创建第一个月份。
3. 有月份但无数据时：表格显示日期和默认店面，金额为 0。
4. 图片未选择时：扫描按钮不可用或提示先上传图片。
5. 扫描中：显示正在识别。
6. 识别成功且信息完整：显示确认入账结果。
7. 识别成功但信息不完整：显示待补充字段。
8. 疑似重复：显示重复确认。
9. 新店面：显示新增店面确认。
10. 年份已满 12 个月：提示当前年份已满。
11. 识别日期对应月份不存在时：提示先新增月份，第一版不自动创建。

## 15. 安全和确认规则

以下操作必须先提醒用户确认：

1. 跨月份写入单据。
2. 新增识别到的新店面。
3. 疑似重复仍确认加入。
4. 取消本次未入账识别结果。
5. 以后如果加入删除、清空、覆盖、导出到外部服务、上传真实客户图片到外部识别服务，都必须先确认。

第一版不做删除、清空、付款、授权、发送消息。

如果用户提供客户隐私图片、账号、密码、密钥、银行卡、身份证等重要信息，必须提醒安全风险，并建议以后不要直接发送这类信息。

## 16. 验收标准

第一版控制面板架构完成后，需要验证：

1. 网页能打开。
2. Mac 浏览器布局清楚。
3. 苹果手机宽度下不重叠。
4. 左侧只显示月份相关内容。
5. 年份能切换。
6. 初始月份显示 2026 年 5 月、4 月、3 月、2 月、1 月，5 月在最上面。
7. 增加月份符合规则，2026 年下一次新增 6 月。
8. 最新月份排最上面。
9. 月份表格显示日期、店面和金额。
10. 每个格子默认显示 0。
11. 同一天同店面多张单会累加显示。
12. 点击格子能看加法明细。
13. 上传文件框和扫描图片按钮明显。
14. 图片未选择时不会直接扫描。
15. 识别不清楚字段会留空。
16. 店面、日期、金额和备注能人工修改。
17. 新店面需要确认后才新增列。
18. 疑似重复需要确认后才加入。
19. 年度统计弹窗能显示当前年份各月总金额。

## 17. 推荐实现顺序

后续进入实现前，建议按这个顺序拆任务：

1. 搭控制面板外框：左侧栏、右侧顶部、主区域。
2. 做年份选择、初始月份列表、增加月份规则。
3. 做月表矩阵：日期行、店面列、金额格子、当日合计、月合计。
4. 做单元格明细弹窗。
5. 做上传文件框和扫描按钮的前端状态。
6. 做模拟识别结果确认区。
7. 做不清楚字段留空和待处理提示。
8. 做新店面确认。
9. 做疑似重复确认。
10. 做年度统计弹窗。
11. 做 Mac 和手机浏览器验证。

## 18. 已确认简化规则

1. 第一版不保存原图。
2. 目标月份不存在时不自动创建，由用户通过增加月份按钮创建。
3. 新店面确认后，以后新增月份默认带出。
4. 修改金额后只保留最新金额，页面不显示最后修改时间。
5. 待处理只显示数量或简单提示，不做点开列表。
