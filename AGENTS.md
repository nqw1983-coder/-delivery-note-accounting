# 水果店手写送货单记账项目 Agent 规则

## 项目目标

本项目服务水果店老板，用来扫描或上传手写送货单图片，自动识别店面、日期、总金额，并填写到对应月份、对应日期的金额统计表中。

第一阶段先做网页版，Mac 电脑和苹果手机浏览器都能使用。网页版稳定后，再考虑微信小程序、苹果端、安卓端。

## 核心原则

1. 项目核心是扫描手写送货单记账，不再以语音记账为主。
2. 第一版先识别店面、日期、总金额，不强制识别水果明细。
3. 自动识别是默认功能。
4. 识别结果清楚时，系统自动填写到对应月份表格。
5. 识别结果不清楚时，宁可留空，也不能瞎填，不能出现虚假信息。
6. 日期、店面、金额看不清或不确定时，要提示用户人工填写或修改。
7. 必须保留修改功能，用户可以修改店面、日期、金额和备注。
8. 每条记录必须能查看原图，方便以后复查。
9. 控制面板操作必须简洁明了，不堆入口，不让老板在多个复杂页面里找功能。

## 第一阶段范围

先做网页控制面板，不急着做完整后端。

第一版核心功能：

1. 左侧显示月份列表。
2. 第一版左侧有效月份只显示 2026 年 5 月、4 月、3 月、2 月、1 月。
3. 月份区域保留 12 行高度，其余 7 行留空，不显示文字和状态。
4. 每个页面显示一个月份的金额统计。
5. 左侧底部设置上传文件框、扫描图片按钮和待补提示，并和页面下端对齐。
6. 上传手写送货单图片后，自动识别店面、日期、总金额。
7. 识别清楚的信息自动填入表格。
8. 识别不清楚的信息保留空白，并提示人工填写。
9. 支持修改已填写的店面、日期、金额和备注。
10. 支持按日期查看当天所有送货单。
11. 保留原图和修改记录，方便追溯。
12. 识别不清楚的单据可以当场补充，也可以进入待处理状态，但不在左侧单独增加待处理入口。

## 页面结构

网页端采用控制面板结构：

1. 左侧：月份列表、上传入口、扫描图片按钮、待补提示。
2. 顶部：当前月份。
3. 中间：本月金额统计表。
4. 右侧或弹窗：单据详情、识别结果、修改记录。

左侧只放月份相关内容，不额外放待处理入口。待处理数量只在左侧底部上传扫描区用简短提示显示，避免左侧变复杂。

月份表格建议字段：

| 日期 | 店面数量 | 当日总金额 | 状态 | 操作 |
|---|---:|---:|---|---|

当天单据明细建议字段：

| 店面 | 日期 | 金额 | 原图 | 状态 | 操作 |
|---|---|---:|---|---|---|

## 图片识别规则

手写送货单识别优先级：

1. 先识别日期。
2. 再识别总金额。
3. 再识别店面或客户名。
4. 识别清楚才填写。
5. 识别不清楚必须留空，不能猜。
6. 没有日期时，不自动归入具体日期。
7. 没有金额时，不自动生成金额。
8. 同一天、同店面、金额接近时，要提示可能重复。

样例单据特点：

1. 日期通常在右上角。
2. 总金额通常在右下角或合计栏。
3. 店面或客户名通常在左上角或表格上方。
4. 手写内容可能不清楚，所以必须允许人工修改。

## 数据要求

每条入账记录至少保存：

1. 店面。
2. 日期。
3. 所属月份。
4. 总金额。
5. 原始图片。
6. 自动识别结果。
7. 人工修改后的结果。
8. 创建时间。
9. 修改时间。
10. 备注。

金额统计必须能追溯到原始图片和修改记录。

## 安全边界

涉及以下操作时，必须先提醒用户确认：

1. 删除图片或账单。
2. 清空月份数据。
3. 覆盖已有记录。
4. 强制推送代码。
5. 付款、扣费、购买服务。
6. 授权第三方账号或读取隐私数据。
7. 上传包含客户隐私的图片到外部服务。
8. 发送消息、邮件、通知给外部人员。

用户发送账号、密码、密钥、银行卡、身份证、客户隐私等重要信息时，要提醒安全风险，并建议以后不要直接发送这类信息。

## 开发流程

每个任务按 ODD 方式推进：

1. 目标：说明这一步解决什么问题。
2. 拆分：拆成 Claude Code 能实现的小任务。
3. 交付：说明本次交付哪些页面、表格、按钮或数据能力。
4. 检查：Codex 严格审查实现结果。
5. 验证：运行页面，测试关键流程，不能只看代码。
6. 记忆：记录已完成内容、重要决定、问题和下一步。

复杂任务不能直接开做，要先讨论思路、范围和优先级。

## Codex 和 Claude Code 分工

Codex 负责：

1. 制定整体设计规划。
2. 拆分任务。
3. 审查 Claude Code 的实现。
4. 验证页面和流程。
5. 检查是否有虚假完成。
6. 记录阶段记忆。
7. 遇到风险时提醒用户。

Claude Code 负责：

1. 按任务写代码。
2. 说明实现内容。
3. 提供测试结果。
4. 说明未完成或不确定的地方。
5. 不得把未验证的功能说成已完成。

Codex 审查重点：

1. 页面是否适合水果店老板使用。
2. 月份列表和金额表格是否清楚。
3. 上传图片和扫描按钮是否明显。
4. 识别清楚的信息是否正确填入。
5. 识别不清楚的信息是否留空，没有瞎填。
6. 店面、日期、金额是否可以修改。
7. 修改后统计金额是否更新。
8. 是否能查看原图追溯。
9. 是否做过实际运行验证。

## 验证标准

每次完成后，尽量实际验证：

1. 页面能打开。
2. Mac 浏览器能操作。
3. 苹果手机浏览器布局不乱。
4. 月份列表能切换。
5. 图片能上传。
6. 扫描按钮能触发识别流程。
7. 识别结果能显示。
8. 不清楚的识别项会留空。
9. 店面、日期、金额能修改。
10. 确认后能写入对应月份。
11. 同一天多张单据能查看明细。
12. 修改后金额统计会更新。

无法验证时，必须说明原因和风险，不能说已经完成。

## 卡死停止规则

遇到以下情况必须停止，不得继续循环或反复重试：

1. 同一工具或命令相同参数连续失败 2 次。
2. 连续 3 步没有实质进展。
3. 遇到删除、清空、覆盖、强制推送等不可逆操作。
4. 关键工具不可用且无法确认原因。
5. 图片识别结果无法验证。

停止时必须说明：

1. 卡在哪一步。
2. 已经完成了什么。
3. 建议用户下一步怎么做。

## 插件和工具使用

有合适工具时要主动使用，但必须服务项目目标。

优先使用：

1. 浏览器工具：检查网页是否真的能打开和操作。
2. 图片识别能力：验证手写送货单识别效果。
3. 表格工具：验证月份金额统计和导出。
4. 文档工具：整理需求、验收标准、任务说明。
5. 记忆工具：记录阶段成果和重要决定。

不能把没有验证过、无法调用、只是猜测可用的能力说成已经可用。

## 记忆规则

每完成一个阶段，要记录：

1. 已完成什么。
2. 当前页面状态。
3. 当前功能状态。
4. 重要设计决定。
5. 未完成事项。
6. 下一步建议。

记忆要简洁，方便换窗口后继续推进。
